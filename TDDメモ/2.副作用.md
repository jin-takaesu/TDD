# 副作用

---

   Green でテストコードは通るが、通った後に問題が発覚することがある。
今回起きていることは関数やメソッドが呼び出し元のデータを変更してしまっていることです。
この影響を**副作用**という

---

## 現在の Dollar.h

```cpp
class Dollar {
public:
    int amount;                              // 型を用意（int）
    Dollar(int amount) : amount(amount) {}   // コンストラクタ
    void times(int multiplier) {             // 掛け算関数
        amount *= multiplier;
    }
};
```

## 現在の test.cpp

```cpp
TEST(DollarTest, Multiplication) {
    Dollar five(5);
    five.times(2);
    EXPECT_EQ(10, five.amount);
}
```

---

この状態。テストコードは通るが、一点副作用がある。

```cpp
void times(int multiplier) {
    amount *= multiplier;
}
```

この `times` 関数の中身 `amount *= multiplier;` という部分。
やっていることとしては `amount`、今は5という数字が入っていてそこに2をかけて10にする。
その10を再度 `amount` に入れる。つまり `amount` にある数字の内容を更新するイメージ。

一見普通に計算してるじゃないか。何がいけないんだ！と言いたくなるが、
これだと呼び出し元の `amount` の中身が変わり期待値が予測できなくなるという副作用があり、
今回の場合は `five` という変数名なのに2をかけて10にしてしまうとこの変数は使いづらくなります。

```cpp
Dollar five(5);

five.times(2);  // five.amount が 5 → 10 に変わる
five.times(3);  // 10 × 3 = 30（期待は 5 × 3 = 15）
```

上記のようにテストコードが増えていくと「今の変数の値なんだっけ？」となりややこしい感じになります。
他にも

- 同じ値を使い回せない
- テストの順番に依存してしまう
- バグの原因が追いにくい

などがあり大変厄介。

---

## 目標

**「テストを正しく実行且つ元の値を変えない」**

---

## Step 1: Red — テスト側を修正する

まずは TDD なのでテスト側を修正します。

```cpp
Dollar five(5);
Dollar ten = five.times(2);      // times() が Dollar を返すことを要求
EXPECT_EQ(10, ten.amount);
Dollar fifteen = five.times(3);  // five は 5 のまま
EXPECT_EQ(15, fifteen.amount);
```

この状態でコンパイルすると `times()` が `void` なので `Dollar` に代入できないのでエラーになります。
テストコード側の `Dollar ten = five.times(2);` と Dollar 側の `void times(int multiplier) {}` 、
この二つの言ってることが違うみたいな感じでエラーになります。方向性の違いで解散するバンドみたいなもんです。

## Step 2: Green — Dollar.h を修正する

解散しないように追記したテストコードに合わせて Dollar ファイルを変えます。

```cpp
Dollar times(int multiplier) {
    return Dollar(amount * multiplier);
}
```

こう書くことで `void` から `Dollar` 型になりデータのやり取りができるようになります。
ビルドするとテストが通り Green になります。


これで副作用を取り除くことができ、安心安全なコードになりました。
