# TDD（テスト駆動開発）とは

実装コードを書く**前に**テストを書く開発手法です。Kent Beck によって広められました。

---

## 基本サイクル：Red → Green → Refactor

1. **Red** ：失敗するテストを書く（まだ実装がないので当然失敗）
2. **Green** ：テストが通る最小限のコードを書く
3. **Refactor**：コードをきれいにする（テストは引き続き通る）

---

## C++ での例

内容としては「5 × 2 = 10 か確かめる」というもの。

### テストファイル作成 → `test.cpp`

```cpp
#include "Dollar.h"

TEST(DollarTest, Multiplication) {
    Dollar five(5);
    five.times(2);
    EXPECT_EQ(10, five.amount);
}
```

---

### 1. Red：テストを実行して失敗を確認する

上記のコードで通るか確認する。

**結果：失敗**
そもそも `#include` で指定している `Dollar.h` というファイルが存在しないため。

```
fatal error: Dollar.h: No such file or directory
```

---

### 2. Green：テストを通す実装を追加する

#### ファイル作成 → `Dollar.h`（クラスだけの状態）

```cpp
#include "Dollar.h"　//対象ファイル読み込み
#pragma once         //おまじない

class Dollar {

};
```

この状態でテストを実行するとまたエラーになるが、エラーメッセージに足りないものが表示される。

```
error: no matching function for call to 'Dollar::Dollar(int)'
error: 'class Dollar' has no member named 'times'
error: 'class Dollar' has no member named 'amount'
```

これをもとにさらに追記する。

#### ファイル追記 → `Dollar.h`

```cpp
#include "Dollar.h"
#pragma once

class Dollar {
public:
    int amount;                              // 型を用意（int）
    Dollar(int amount) : amount(amount) {}   // コンストラクタ
    void times(int multiplier) {             // 掛け算関数
        amount *= multiplier;
    }
};
```

これでテストコードが通るようになります。

> エラーメッセージが「追記すべきもの（TODO）」を示してくれる。
> それをリスト化して実装したものからチェックを入れていき、テストコードが通るように実装していくのが
> **Red** と **Green** における大まかな流れです。
>
