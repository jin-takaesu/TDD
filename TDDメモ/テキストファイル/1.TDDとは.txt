
TDD(テスト駆動開発)とは
実装コードを書く前にテストを書く開発手法です。Kent Beck によって広められた


基本サイクル：Red → Green → Refactor
---------------------------------------------------------------
1. Red    : 失敗するテストを書く（まだ実装がないので当然失敗）
2. Green  : テストが通る最小限のコードを書く
3. Refactor: コードをきれいにする（テストは引き続き通る）
---------------------------------------------------------------


C++での例、内容としては5×2＝10なのか確かめるみたいなもの

---------------------------------------------------------------
testファイル作成→test.cpp

#include "Dollar.h"

TEST(DollarTest, Multiplication) {
    Dollar five(5);
    five.times(2);
    EXPECT_EQ(10, five.amount);
}
---------------------------------------------------------------
1. Red　：上記のコードで通るか確認する。
　→結果：失敗(それはそう。そもそもincludeで指定している「Dollar.h」なんてファイルはないのだから)
　　→エラーメッセージに内容はfatal error: Dollar.h: No such file or directory

2. Green　：以下のコードを追加(ファイル作成)して通るようにする。

---------------------------------------------------------------
ファイル作成→Dollar.h

#pragma once

class Dollar {

};
---------------------------------------------------------------

この状態(クラスだけ)でテスト実行するとまたしてもエラーになるがエラーメッセージに

error: no matching function for call to 'Dollar::Dollar(int)'
error: 'class Dollar' has no member named 'times'
error: 'class Dollar' has no member named 'amount'

と足りないものがエラーメッセージに出力される。それをもとにさらに追記

---------------------------------------------------------------
ファイル追記→Dollar.h

#pragma once

class Dollar {
public:
    int amount;　　　　　　　　　　　　　　　//型を用意(int)
    
    Dollar(int amount) : amount(amount) {}   //コンストラクタ

    void times(int multiplier) {　　　　　　 //掛け算関数
        amount *= multiplier; 
    }
};
---------------------------------------------------------------

これでテストコードが通るようになります。
このようにエラーメッセージが追記すべきものを示してくれます（TODO）
これをリスト化して実装したものからチェックを入れていきテストコードが通るように実装していくのが
「1. Red」と「2. Green」における大まかな流れです。
今回書いたものはわかりやすくするためシンプルなものにしたので3のリファクタリングは次章から説明。


最後にこの開発方法のメリットの説明

メリット			説明
バグの早期発見			実装と同時にテストが走るため
設計の改善			テストしやすい設計 = 疎結合な設計
リファクタリングの安全性	テストがあるので変更を恐れなくていい
ドキュメント代わり		テストコードが仕様を表す
